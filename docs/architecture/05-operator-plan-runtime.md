# Operator Plan Runtime v1

## Purpose

The Operator Plan Runtime converts a contract-bound plan into a sealed sequence of approved professional operators and independently checks whether an execution trace actually followed that sequence.

A planner may propose an Operator Plan, but it cannot certify the plan or its execution. Compilation and trace verification are deterministic Rust operations outside the language-model trust boundary.

## Operator Plan

Operator Plan schema `0.1.0` contains:

- an immutable plan identifier;
- exact Work Contract and Profession Capsule SHA-256 bindings;
- ordered plan steps;
- operator identifiers and locked versions;
- explicit step dependencies;
- declared input and output artifact identifiers;
- allowed capability-token identifiers;
- mandatory or optional step status;
- optional rollback-step references.

## Compilation invariants

A plan is rejected unless all of the following hold:

1. The schema version is supported.
2. Contract and capsule bindings use SHA-256.
3. Contract ID and digest match the compiled Work Contract.
4. Capsule ID and digest match the loaded Profession Capsule.
5. Every step ID and sequence is unique.
6. Sequences are contiguous from zero.
7. Every operator is allowed by the selected job type.
8. Every operator version matches the sealed capsule registry.
9. Every dependency exists and points to an earlier step.
10. Every rollback-step reference exists.
11. Per-step dependency, artifact and capability lists contain no duplicates.
12. At least one mandatory step exists.

The complete plan receives a canonical SHA-256 digest after these checks pass.

## Trace state machine

Each step begins in `NOT_STARTED`. The normal successful path is:

```text
NOT_STARTED -> STARTED -> SUCCEEDED
```

Other terminal states are `FAILED`, `ROLLED_BACK` and `SKIPPED`. Invalid transitions are recorded as trace violations.

Before a step can start:

- every declared dependency must already be `SUCCEEDED`;
- the event operator must match the sealed plan step;
- the event must present a capability token authorized for that step when tokens are required.

After all events are processed, every mandatory step must be `SUCCEEDED`. A started-but-unfinished, failed, rolled-back or skipped mandatory step prevents conformance.

## Independent conformance

Evidence Bundle schema `0.3.0` renames `conforms_to_plan` to `claimed_conforms_to_plan` and requires every trace event to carry a `step_id`.

The Evidence Runtime now:

1. verifies the bundle's Operator Plan ID and digest;
2. decodes typed trace events;
3. recomputes trace conformance with the Operator Plan Runtime;
4. rejects a false conformance claim;
5. rejects an honestly reported nonconforming trace as unacceptable work;
6. admits proof results only after the trace is independently proven conformant.

A correct output cannot therefore compensate for an unauthorized or incomplete execution path.

## Current violation classes

The runtime reports distinct violations for:

- duplicate event identifiers;
- non-contiguous event sequence numbers;
- unknown plan steps;
- operator mismatch;
- incomplete dependencies;
- invalid state transitions;
- missing capability tokens;
- unauthorized capability tokens;
- failed or rolled-back steps;
- skipped mandatory steps;
- mandatory steps that never succeeded;
- started steps that never reached a terminal state.

## Trust boundary

Untrusted inputs include:

- plans generated by language models;
- executor success messages;
- event order supplied by an application bridge;
- bundle-declared trace conformance;
- free-form deviation descriptions.

Trusted outputs are limited to the compiled plan digest and the deterministic `TraceAssessment` produced by this runtime.

## Deferred capabilities

Future versions will add:

- signed capability tokens with resource scopes and expiry;
- artifact-flow verification between plan steps;
- authorized deviation plans;
- rollback subgraphs instead of simple references;
- concurrency groups and deterministic synchronization barriers;
- trace signatures and trusted timestamps;
- model-checking adapters for high-assurance workflows.
